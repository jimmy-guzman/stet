# Code intelligence

- Beyond the push-diagnostics poll, sideye makes on-demand read-only `textDocument/*` pull requests over the same warm `LanguageServers` pool (`src/intel/`). A pull is a `didOpen` → request → `didClose` bracket on the first acquired server that advertises the matching capability, parsed from the server's `initialize` reply into a capability set on the `ServerHandle` (so oxlint, which advertises none, drops out and `typescript-language-server` answers). Most pulls send one request; call hierarchy sends two (a `prepare` then a resolve) inside the one open document, sharing a single project-load wait and one close. Reads only, exactly like `publishDiagnostics`: no edit, format, or rename request is ever sent.
- A location reply carries absolute `uri`s; the service relativizes each to a repo path (reusing `relativize`, the same boundary the diagnostics checker uses). A target outside the repo (a definition in `node_modules`) stays absolute, and the caller skips it rather than opening an out-of-tree path. The pull bracket is generic over the reply shape: the location callers relativize inside their own `normalize`, while hover returns plain text and skips that step.
- A pull shares the pooled connection with the diagnostics flow, so its open/close bracket could race a concurrent open of the same document (a poll mid-settle, or a second pull). The transport refcounts each document per `uri` (`openDocument`/`closeDocument`): `didOpen` is sent only for the first holder and `didClose` only when the last holder releases, so a second open no longer resets the server's view of a doc another holder still needs, nor drops one out from under it. Both the intel bracket and the diagnostics open/settle/close go through it.
- A pull waits for the server's project to finish loading before it requests. The client advertises `window.workDoneProgress` in `initialize`, the transport tracks the project-load `$/progress` lifecycle, and `whenProjectLoaded` resolves on the first `end` (or on connection close). Until then a cold `typescript-language-server` answers `textDocument/definition` from the local import binding instead of resolving cross-file, so F12 on an import would stop at the import line; the gate is what makes a cold/slow server (a large monorepo) return the real source instead. The wait is interruptible (the caller aborts on the next navigation) and capped by a generous backstop, and it is a no-op once the project is warm.
- Go to definition (`F12`) resolves the symbol under the caret and jumps via the existing `jumpTarget` path (a cross-file jump pushes history, so `<` returns to the call site); a single in-repo target jumps, and more than one (an overloaded symbol) opens the references overlay labelled `definitions` rather than a blind jump to the first (picking a result with `enter` or a click jumps via `jumpTarget` and dismisses the overlay). A request failure degrades to a notice, never an exception into the UI. While the pull is in flight the status bar carries an in-flight indicator (`resolving definition…`, an `info`-level status), set on the keystroke and cleared when the pull settles (jump, notice, or a superseding request), so the wait for a cold server never reads as a dead keypress.
- Find references (`Shift+F12`) resolves every use of the symbol under the caret (`textDocument/references` with `includeDeclaration`) and opens the query-less references overlay: results grouped by file, each row a tabular `path:line:col` plus the trimmed source line; picking a result with `enter` or a click jumps via the same `jumpTarget` path and dismisses the overlay, while `esc` closes it without navigating. It guards exactly like definition, opens at once in a loading state, and resolves in place to the list, an empty screen (`no references`), or an error, so the wait never reads as a dead keypress without borrowing the status bar. Each result's source line is read locally for the preview (a missing or binary file still lists the row, without one); go-to-definition's multi-result case reuses the same overlay.
- Find implementations (`Shift+I`) resolves the concrete bodies of the symbol under the caret (`textDocument/implementation`) and behaves exactly like go-to-definition: the reply shape is identical (`Location | Location[] | LocationLink[]`), so it reuses `normalizeDefinition` and the same single-jump-vs-multi-overlay path. A single in-repo target jumps via `jumpTarget`; more than one opens the references overlay labelled `implementations` (a non-directional overlay, so `Tab` is a no-op). It guards exactly like definition, carries the same in-flight status indicator (`resolving implementations…`), and a failure degrades to a notice, never an exception. It is distinct from definition: definition lands on the abstract declaration, implementation jumps past the abstraction to the concrete bodies (an interface/abstract member lists every implementor; a plain concrete symbol collapses to a single jump). Client and server must both advertise the capability (`implementationProvider`, gated on the `implementation` client cap declared in `initialize`).
- Call hierarchy (`Shift+H`) lists the callers of the function or method under the caret in the same references overlay. It is a two-step pull: `textDocument/prepareCallHierarchy` returns an opaque item (its server-defined `data` preserved), then `callHierarchy/incomingCalls` (who calls this) or `callHierarchy/outgoingCalls` (what this calls) resolves the edges, both inside the one open document with one project-load wait, closing once. `Tab` flips the direction and re-resolves the same symbol (superseding the prior direction via the shared intel controller); the overlay label and footer say which way you're looking, and a plain references/definitions overlay (no direction) leaves `Tab` a no-op. It guards exactly like definition (caret on a symbol, on a line that exists in the file), opens at once in a loading state, resolves in place to the list / an empty screen / an error, and normalizes each caller or callee to its name range so a row jumps via the same `jumpTarget` path. Client and server must both advertise the capability; a server without it (`typescript-language-server` advertises `callHierarchy` but not type hierarchy) yields the empty state. Type hierarchy reuses this exact two-step machinery and is a deliberate follow-up, gated on a server that provides it.
- Hover (`K`) shows type + docs for the symbol under the caret in a caret-anchored card (`textDocument/hover`). The reply is parsed into ordered code/prose segments (the fence delimiters dropped): each code block is syntax-highlighted in its language through the same shared Shiki highlighter and active theme the diff uses (so the signature matches the diff's colors and re-themes with it), and prose renders plain. It guards exactly like definition (caret on a symbol, on a line that exists in the file) and degrades to an empty/error card, or to uncolored code when a language can't be highlighted, rather than throwing. Keyboard-only in v1; pointer-hover is a deliberate follow-up.
- Find symbols (`S`) lists the open file's symbols (`textDocument/documentSymbol`) in a query-less palette-family overlay, each row a kind glyph (a Nerd Font codicon gated on `iconsEnabled`, a text tag otherwise) plus the symbol name and its `line:col`, indented to mirror the file's structure. Unlike the caret pulls it needs only an open file (the whole document is the query), so it guards on `selectedPath` alone; picking a symbol with `enter` or a click jumps via the same `jumpTarget` path (always within the open file, escalating to full-file view when the line is outside the diff) and dismisses the overlay. It opens at once in a loading state and resolves in place to the list, an empty screen (`no symbols`), or an error. The client advertises `hierarchicalDocumentSymbolSupport` in `initialize`, so a server returns the nested `DocumentSymbol[]` (flattened pre-order into a depth-tagged, position-ordered outline); a server that still replies with a flat `SymbolInformation[]` degrades to a depth-0 list. Its own controller (not the shared definition/references one, since it owns a separate overlay), and it closes on repo or file drift.
