# Content search

- `ctrl-f` opens the project search pane, the first **main-area view mode**: a `mainView` union signal (`"file" | "search"`) swaps the Viewer's interior, so exactly one view is ever active and future panes extend the union rather than adding booleans. The pane is persistent: closing it (`esc`, or any navigation to a file, including a result jump) keeps the query, results, selection, and collapsed groups, so `ctrl-f` after a jump restores the result set instead of clearing it. A worktree switch clears the snapshot (stale paths) but keeps the query.
- The engine is `git grep` (`-I` to skip binary, `--untracked` so it covers the tree's universe, `--column` for the match column). It searches working-tree file content, not diffs. Literal `-F` with smart-case by default; a regex toggle (`ctrl-r`) swaps `-F` for extended regex `-E`, and a case toggle (`ctrl-x`) forces case sensitivity over smart-case. `--column` reports a byte offset; it is converted to UTF-16 units so a jump lands the caret on the exact column even on non-ASCII lines.
- Scope defaults to the changed set (pathspec-limited to `git status`'s changed files, so it follows the active all/staged/unstaged scope); `ctrl-g` toggles between the changed set and the whole repo, and `ctrl-s` opens the scope picker without leaving the pane; the scope cell names the effective universe (`repo`, or the active diff scope: `changes`, `staged`, `unstaged`, `session`, `last commit`). A filter field narrows by pathspecs (space-separated, git wildmatch): plain tokens include, a `!` prefix excludes (mapped to git's `:(exclude)`, so includes union and excludes subtract; an exclude-only filter matches everything else, and raw pathspec magic passes verbatim). Under the changed scope the filter drives the grep and the changed set filters the results after, since include pathspecs union rather than intersect.
- Results group by file with per-file headers (match count, collapsible with `enter`/`h`/`l`), two lines of context around each match (overlapping context ranges merge into one excerpt, `⋯` separates non-contiguous excerpts), and Shiki syntax highlighting per file language through the diff's shared highlighter (highlighted on demand for the visible window; rows render plain and upgrade in place). Context comes from bounded file reads, not `git grep -C` (whose `-z` output makes match and context records indistinguishable); an unreadable or oversized file degrades to match-only rows. `enter` routes through the same `jumpTarget` path as problems navigation, landing on the line and column and escalating to full-file view when the line is outside the diff.
- Keyboard while the pane is focused: `tab` cycles query → glob → results, `↓` enters the results, `j`/`k` walk navigable rows (matches and headers; context and gaps are skipped), `ctrl-d`/`ctrl-u` half-page by _viewport_ rows (snapping to the nearest navigable row). Only the query/glob inputs swallow printable keys, and they keep their readline chords (`ctrl-a`/`ctrl-e` line home/end, `ctrl-w`/`ctrl-u`/`ctrl-k` kill); `ctrl-p` (palette) and the deliberately stolen `ctrl-b` (sidebar, with preventDefault) pass through even there. In results focus unhandled keys fall through, so the global bindings (`q`, `?`, `ctrl-p`, `p`...) keep working, matching the problems pane. A single click on a result selects it (a focus-intent click never navigates the view away); a double click, like `enter`, opens it. `esc` closes panels innermost-first: the problems panel, then the search view, then the app. File-view keys (`v`, `f`, `z`, `ctrl-t`/`ctrl-w` tab pin/close, `F12`/`Shift+F12`/`K` intel, `y`/`Y` viewer copy) are inert while the search view shows, since they would act on a viewer that isn't on screen; in the results, `e`/`o` open the selected result in the editor, `y` copies its reference (`path:line:col` on a match), and `g`/`G` jump to the first/last result. Any navigation to a file (a result jump, the palette, problems, recency, history) reveals the file view, including a jump that lands on the already-selected file. The chrome rows (query, filter, summary, footer) are fixed-height in every state, so idle/searching/no-matches/error swaps never shift layout.
- The query is debounced and runs through the interruptible `Process` service, holding the previous snapshot until the new one resolves; a failed grep (a half-typed regex) keeps the prior results under an error notice. While the pane is open the search re-runs on every git-model refresh (and on reopen), so results, context lines, and jump targets track the working tree as an agent edits it. Results are capped to keep the pane bounded; hitting the cap is surfaced as a trailing `+` on the count.
