Configuration is optional. Without it, stet uses its built-in `dark` or `light`
theme to match your terminal and reads editor commands from environment
variables, with `vim` as the terminal-editor fallback.

## Location

stet reads its config once at startup from:

```
~/.config/stet/config.jsonc
```

`$XDG_CONFIG_HOME` is honored, and `config.json` works too. Both are parsed as
JSONC, so comments and trailing commas are allowed.

<Callout>
  A malformed or invalid config never blocks startup: stet falls back to defaults and shows a
  notice.
</Callout>

## Editor and IDE

Use `editor` and `ide` to set persistent command templates for the `e` key
(terminal editor) and the `o` key (GUI / IDE). Both use `{file}` and `{line}` as
placeholders; `{line}` is omitted automatically when no cursor line is
available. The `ide` template also takes `{repo}` (the repo root), which passes
the folder to the IDE so it opens the file as a workspace instead of a bare
single-file window.

```jsonc title="~/.config/stet/config.jsonc"
{
  "editor": "nvim +{line} {file}",
  "ide": "code {repo} --goto {file}:{line}",
}
```

Without a config value, each key falls back to `STET_EDITOR` / `STET_IDE`, then
`$EDITOR` / `$VISUAL`, then `vim` (editor only); `o` does nothing if nothing is
configured. A bare editor name with no `{file}` is expanded to a known template
(`nvim` becomes `nvim +{line} {file}`, `code` becomes `code {repo} --goto
{file}:{line}`, and so on), so the known GUI IDEs (`code`, `cursor`, `zed`,
`idea`) open a workspace out of the box. Templates are split on whitespace, so an
editor binary path with spaces is not supported.

The `O` key opens the current file in your OS default app (`open` on macOS,
`xdg-open` on Linux). It uses the system handler, so it needs no template or
config.

## Themes

Define themes under `themes` and pick one with `theme`. `theme` is either a
single theme name, or a `{ "dark": ..., "light": ... }` pair that follows the
terminal live: flip your terminal's appearance and stet re-themes.

A theme is a full set of `#rrggbb` tokens, or `{ "base": <name>, ... }` that
inherits another theme and overrides only the tokens you name. Its `"syntax"` is
a bundled [Shiki](https://shiki.style/) theme name, or an object overriding
individual tokens (`keyword`, `string`, and so on).

```jsonc title="~/.config/stet/config.jsonc"
{
  // follow the terminal, with a custom theme on each side
  "theme": { "dark": "my-dark", "light": "my-light" },
  "themes": {
    "my-dark": { "base": "dark", "accent": { "primary": "#ffa7d9" } },
    "my-light": { "base": "light", "accent": { "primary": "#b4267a" } },
    "mocha": { "base": "dark", "syntax": "catppuccin-mocha" }, // stet chrome, Catppuccin code
    "tweaked": { "base": "dark", "syntax": { "keyword": "#ff8800" } }, // one token changed
  },
}
```

The built-in themes are `dark` and `light`. Press `t` in the app to open the
theme switcher and try any theme without editing the config: filter by name,
move or hover to preview the whole UI live, `enter` (or click) to apply, `esc`
to revert. The switch lasts the session; the config file is where a theme is
made permanent.

## Languages

Use `languages` to add languages backed by your own language servers, or to
change which servers run for a built-in language. Diagnostics and code
navigation then work for those files exactly as they do for the built-ins.

A language entry declares the file types it owns (`extensions`, and `filenames`
for exact names like `Dockerfile`) and the `servers` that analyze them. A server
entry is written one of these ways:

| Form                                   | What it is                                                  |
| -------------------------------------- | ----------------------------------------------------------- |
| `"biome"`                              | a built-in, by name; it runs                                |
| `{ "server": "biome", "when": [...] }` | a built-in, run only in repos matching `when`               |
| `{ "command": [...], "when"?: [...] }` | any command on your machine, optionally gated the same way  |
| `{ "firstOf": [entries...] }`          | candidates in preference order; the first that applies runs |

The built-in names are `typescript`, `oxlint`, `biome`, `json`, `yaml`,
`rust-analyzer`, `basedpyright`, `ty`, and `ruff`. An inline command also takes
`initializationOptions` and `settings`, passed to the server as-is (`{repoRoot}`
and `{repoUri}` in string values are replaced per repository).

This is the same grammar stet's own built-in table is written in: Biome ships
with `"when": ["biome.json", "biome.jsonc"]`, and Python's server list is
`[{ "firstOf": ["ty", "basedpyright"] }, "ruff"]`. There is no gate a built-in
can express that your config cannot.

```jsonc title="~/.config/stet/config.jsonc"
{
  "languages": {
    // add a language: file types + at least one server
    "lua": {
      "extensions": ["lua"],
      "servers": [{ "command": ["lua-language-server"] }],
    },
    // route exact filenames
    "docker": {
      "filenames": ["Dockerfile"],
      "servers": [{ "command": ["docker-langserver", "--stdio"] }],
    },
    // change a built-in: unset fields are inherited, "servers" replaces the
    // list, so leaving oxlint out turns it off for the JS/TS family
    "typescript": { "servers": ["typescript", "biome"] },
    // pick Python's type checker yourself instead of letting stet detect it
    "python": { "servers": ["ty", "ruff"] },
  },
}
```

Naming `servers` turns detection off for that language. Some built-in servers
normally activate only where a repo opted into them: Biome needs a `biome.json`,
and Python's type checker is picked from what the repo declares. That detection
exists to guess what you want, so once you say it, stet stops guessing. Name
`biome` and it runs with or without a `biome.json`; name a Python type checker
and it runs regardless of what the repo declares. Leave `servers` out (to add a file type,
say) and the language keeps its built-in list, detection included.

### Run a server only in some repos

`when` is how you say it yourself. This config is global, so it applies to every
repo you open, and a server with nothing to gate it runs everywhere, reporting
findings in projects that never asked for it. A `when` is one condition or an
any-of list of them, and a condition takes one of three shapes:

| Condition                                          | Holds when                                         |
| -------------------------------------------------- | -------------------------------------------------- |
| `"biome.json"`                                     | that path exists at the repository root            |
| `{ "file": "...", "key": ["a", "b"] }`             | that key path exists in a TOML or JSON(C) manifest |
| `{ "file": "pyproject.toml", "dependency": "ty" }` | a pyproject declares that dependency anywhere      |

```jsonc title="~/.config/stet/config.jsonc"
{
  "languages": {
    "typescript": {
      "servers": [
        "typescript", // always
        { "server": "biome", "when": ["biome.json", "biome.jsonc"] }, // where the repo opted in
        { "command": ["house-lsp", "--stdio"], "when": [".houserc"] }, // yours, only where it applies
      ],
    },
  },
}
```

`key` is an array of segments (manifest keys may contain dots), and the file is
parsed as TOML or JSONC by its extension, so npm dependencies are reachable as
keys: `{ "file": "package.json", "key": ["devDependencies", "@biomejs/biome"] }`.
`dependency` exists because pyproject declares dependencies as requirement
strings (`"ty>=0.0.58"`) that need name parsing; it is supported for
`pyproject.toml` only. A missing or malformed manifest simply means the
condition is not met.

### Pick one of several servers

`firstOf` runs exactly one of its candidates: the first whose condition holds.
Inside the group, a bare built-in name keeps its own gate (the group exists to
pick by condition), and a candidate with no condition always applies, which
makes it the fallback. This is how Python's type checker works out of the box,
and how you would slot in a third:

```jsonc title="~/.config/stet/config.jsonc"
{
  "languages": {
    "python": {
      "servers": [
        {
          "firstOf": [
            { "command": ["mypy-ls"], "when": ["mypy.ini"] }, // yours, where its config is
            "ty", // ty's own gate: ty.toml, [tool.ty], or a declared dependency
            "basedpyright", // no condition: the fallback
          ],
        },
        "ruff",
      ],
    },
  },
}
```

Built-in server names use stet's pinned definitions and standard [server
discovery and provisioning
flow](/docs/reference/languages#how-servers-are-found).

An inline `command` can run any executable on your machine. stet resolves it
from the repo's own `node_modules/.bin`, then your `PATH`, or uses an absolute
path as-is. Examples include `gopls` installed with `go install`,
`lua-language-server` installed with `brew`, and `solargraph` installed as a
gem. stet never downloads an executable for an inline command.

stet reads inline server entries only from your global config. It does not read
server configuration from the repository being inspected, although a configured
executable may resolve from that repository's `node_modules/.bin`.

<Callout>
  An entry that does not validate is reported in a notice and skipped; the rest of the config still
  applies.
</Callout>

## Environment variables

| Variable               | Effect                                                                       |
| ---------------------- | ---------------------------------------------------------------------------- |
| `STET_EDITOR`          | fallback command template for the `e` key                                    |
| `STET_IDE`             | fallback command template for the `o` key                                    |
| `STET_NO_LSP_DOWNLOAD` | do not auto-download a missing language server (same as `--no-lsp-download`) |
| `XDG_CONFIG_HOME`      | base directory for the config file                                           |
